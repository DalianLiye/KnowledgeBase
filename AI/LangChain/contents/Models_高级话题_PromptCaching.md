# Prompt Caching

许多模型厂商都提供提示词缓存（Prompt Caching）功能，用于在重复处理相同tokens时，降低延迟、减少成本

Prompt Caching分两种：
- **隐式缓存**\
  如果请求命中缓存，厂商会自动节省成本，无需开发者手动配置
- **显式缓存**\
  厂商允许手动指定缓存点，实现更精细的控制，或保证成本节约

Prompt Caching通常仅在输入token数超过最低阈值时才会生效,具体以厂商文档为准

缓存使用情况会体现在模型响应的usage metadata（使用元数据）中


**Prompt Caching如何节省成本**\
Prompt Caching不是缓存 “答案”，而是缓存“提示词的处理结果”\
模型不会直接返回上次的回答，仍然会重新生成答案，不影响推理质量与随机性

**缓存过程**\
第一次提问（无缓存）：
1) 处理 Prompt（系统提示 + 上下文 + 问题）\
   将文本转为tokens → 送入模型做计算/编码(最费钱、最耗时)
2) 生成回答\
   逐字生成最终答案(成本低、耗时少)

注：\
缓存的是：Prompt的编码结果、模型对 Prompt 的理解\
不缓存：答案、推理过程、输出内容

第二次提问（命中缓存）：
1) 无需重新处理 Prompt，直接复用缓存\
   大幅节省输入 token 费用，同时降低延迟（省钱 + 速度快)
2) 重新生成回答\
   虽然重新生成，但仅计算输出token，成本极低


**Prompt Caching核心适用场景：**\
- RAG 系统（每次都带大量文档）
- 固定长系统提示（重复使用的系统角色设定）
- 多轮对话（历史消息很长）
- 企业知识库、客服机器人（高频重复上下文）
