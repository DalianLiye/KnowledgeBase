# 关于自动聚类
自动聚类是一种Snowflake服务，可按需无缝、持续地管理聚类表的所有重聚类\
自动聚类会使用后台的无服务器计算资源\
定义聚类表后，系统不一定会立即开始重聚类\
只有在聚类表能受益于重聚类操作时，Snowflake才会对其进行重聚类



# 自动聚类的优势


## 简化维护
自动聚类可以自动做如下操作：
- 监控聚类表的状态，如果评估表能受益于重聚类，snowflake会自动进行
- 不需要指定要使用的仓库，Snowflake在后台执行自动重聚类

用户只需要为每表定义一个群集密钥（如适用）就可以了，未来所有的管理维护都由Snowflake负责


## 全方位控制
可随时使用以下命令暂停和恢复群集表的自动聚类功能
```sql
ALTER TABLE ...SUSPEND / RESUME RECLUSTER
```
当表的自动聚类暂停时，无论聚类状态如何，表都永远不会再自动重聚类，因此也不会产生任何 Credit 费用\
同时可以随时删除聚类表上的群集密钥，从而防止将来对该表执行的所有重聚类


## 不会阻止DML
自动聚类是透明的，重聚类期间不会影响DML语句的执行



## 效率高
通过自动聚类，Snowflake会在内部管理聚类表的状态，以及用于所有自动聚类操作的资源（服务器、内存等）\
这允许 Snowflake 按需动态分配资源，从而实现最高效、有效的重聚类\
此外，自动聚类不会执行任何不必要的重聚类\
仅在表能够受益于重聚类操作的情况下，才会在适当的时机触发重聚类



# 为表启动自动聚类
大多数情况下，不需要执行任何任务即可为表启用自动聚类，只需为表定义群集密钥即可\
但此规则不适用于通过克隆 (CREATE TABLE ...CLONE ...) 从具有群集密钥的源表创建的表\
在新表启动时，自动聚类处于暂停状态 – 即便源表的自动聚类未暂停也不例外\
（无论 CLONE 命令克隆的是表、包含表的架构还是包含表的数据库，都是如此。）

以下情况，在启动自动聚类时可能会执行重聚类操作：
- 该表未处于最佳聚类状态
- 表上的群集密钥已更改

因此，我们建议从一两个选定表开始，并评估自动聚类对这些表的影响\
熟悉自动聚类如何执行重聚类后，即可为其他表定义群集密钥

若要为表添加聚类，还必须具备包含该表的架构和数据库的 USAGE 或 OWNERSHIP 权限


# 查看表的自动聚类状态
查看表是否启用了自动聚类，通过以下方式：
- SHOW TABLES命令
- TABLES 视图（在 Snowflake Information Schema 中）。
- TABLES 视图（在 Account Usage 共享数据库中）。

返回结果中，AUTO_CLUSTERING_ON 列显示每个表的自动聚类状态，可以通过该状态决定暂停或恢复表的自动聚类\
此外， CLUSTER_BY 列 (SHOW TABLES) 或 CLUSTERING_KEY 列（TABLES 视图）会显示定义为每个表的群集密钥的列或表达式


# 为表暂停自动聚类
执行命令：
```sql
ALTER TABLE t1 SUSPEND RECLUSTER;
```
更改表的群集密钥会自动恢复启动自动聚类，这会导致启用无服务器资源，从而消耗Credit\
将关键字LINEAR包含在 ALTER TABLE ...CLUSTER BY 语句中被视为对群集密钥的更改，即使列没有更改


# 为表恢复自动聚类
执行命令：
```sql
ALTER TABLE t1 RESUME RECLUSTER;
```
以下情况，在恢复自动聚类后，可能执行重聚类操作
- 该表未处于最佳聚类状态（例如，自上次重聚类以来，对表执行了大量 DML）。
- 表上的群集密钥已更改



# 自动聚类成本
启用自动聚类的成本类型有以下几种：
- 计算成本\
  自动聚类会使用后台的无服务器计算资源\
  Snowflake首次使用无服务器计算资源 对表进行聚类(初始聚类)\
  在向表中添加新数据时，它还会使用计算资源将表保持在良好的聚类状态(按需重聚类)\
  对表做出的更改越多，维护成本越高
- 存储成本\
  由于自动聚类是重新整理现有数据，而不是创建额外的存储，因此在很多情况下不会产生额外的存储成本\
  但是，如果重聚类增加故障安全存储的大小，则会产生额外的存储成本


# 自动聚类的Credit消耗与仓库
自动聚类所使用的仓库是Snowflake内置的，不是用户创建的\
因此，Snowflake会在内部对自动聚类进行高效资源管理

启动自动聚类并不会消耗credit，只有它真的跑起来、用了计算资源（credits），才会实际消耗的 credits

启动自动聚类后，Snowflake会尝试将表置于最佳聚类状态\
在表的聚类达到最优状态后，重聚类活动将停止

在现有表上定义群集密钥或更改聚类表上的群集密钥都可能触发重聚类和Credit 费用

为了防止产生任何意外的 Credit 费用，建议从一两个选定的表开始，并观察在执行 DML 时为保持良好聚类而产生的相关 Credit 费用\
这有助于为重聚类活动消耗的 Credit 建立基线


# 估算自动聚类费用
以下函数可用来估算表自动聚类的计算成本，以及预测更改表的群集密钥的计算成本\
SYSTEM$ESTIMATE_AUTOMATIC_CLUSTERING_COSTS\
注：该函数返回的是成本估算值，实际发生成本值可能会相差很大


# 查看自动聚类成本
自动聚类会使用后台的无服务器计算资源，因此它会消耗Credit\
自动聚类是自动在后台维护的，包括对表的初始聚类和按需重聚类

查看自动聚类成本，可以通过以下方式：
- Web UI (Snowsight/Classic Console)
- SQL查询 (查看视图：AUTOMATIC_CLUSTERING_HISTORY)

资源监视器可以控制虚拟仓库的Credit使用量，但不能控制Snowflake内置仓库的Credit使用量\
Snowflake内置仓库就包括了AUTOMATIC_CLUSTERING仓库，它是专门用于聚类的
