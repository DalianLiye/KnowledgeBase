# 注意事项
高效使用仓库的关键：
- 通过尝试不同类型的查询和不同大小的仓库，以确定最佳匹配组合
- 由于Snowflake采用按秒计费方式，可以选择运行更大的仓库，在使用后马上暂停，不要过度专注仓库大小


# Credit计算
关于credit费用的计算，依据以下几点：
- 仓库大小  (仓库每升1级，其资源数通常会翻倍，费用也翻倍)，比如
  仓库大小是X-small，每个群集连续运行一小时，收取1个credit
  仓库大小是small，每个群集连续运行一小时，收取2个credit
- 集群数量(如果是多集群仓库)
- 每个集群中计算资源的运行时间

**一般情况下的Credit计费规则：**
- 最低计费时间为60 秒，比如资源刚启动就马上关闭，也是计费60秒
- 资源启动60秒后，变成按秒计费，直到所有资源关闭，比如运行了61秒，则按61秒计费

**调整仓库大小时的Credit计费规则：**
- 调整仓库变大，会增加额外计算资源，会使Credit计费相应增加
- 调整仓库变大，增加的Credit计费是仓库大小调整后的时间计算的
- 将 5XL 或 6XL 仓库调整为 4XL 或更小的仓库会导致在旧仓库处于静默状态时，短时间内向客户同时收取新仓库和旧仓库的费用
- Credit使用量以小时为增量显示，在按秒计费时，会以小数形式显示的credit使用/计费数量


# 查询组成与仓库处理
处理查询所需的计算资源，取决于以下几点：
- 查询的大小
- 查询的复杂程度

一般来说，仓库越大，能处理的查询规模就越大，尤其是对于更大、更复杂的查询

影响查询性能和资源消耗，主要有以下因素：
- 被查询表的总体大小
  总体越大，性能和资源消耗越大，总体主要看每行/整体的字节数，而不是单纯的行数
- 使用的谓词(where/join on/having等)
  如果谓词使用很高效，能很好的提升查询性能和减少不必要的资源消耗
- 查询中的连接/表数量
  查询中使用的连接/表数量越多，越影响查询性能和资源消耗


**注：**\
尽量在同一仓库中执行相对同质的查询（复杂程度、数据集等）
在同一仓库中执行大小或复杂程度差异较大的查询会增加分析仓库负载的难度
从而增加选择最佳仓库大小以匹配工作负载中查询的复杂程度、组成和数量的难度


# 仓库缓存
仓库运行时，会缓存查询表的数据，用于提高后续查询性能
仓库越大，能缓存的数据越多

当仓库暂停时，缓存就会被清理
当仓库恢复时，需要重新缓存数据，因此恢复后执行同样的查询会性能降低
仓库保持运行，虽然缓存会一直存在，提高查询的性能，但仓库运行会一直消耗credit，因此需要权衡性能和成本


# 创建仓库
从成本和性能角度来看，创建仓库时需考虑以下关键因素：
- 仓库大小 
- 手动管理与自动管理（用于启动/恢复和暂停仓库）
- 集群数量 （Snowflake Enterprise Edition）


# 选择仓库大小
根据要执行的任务及其处理的工作负载，来选择仓库的大小

针对数据加载操作，仓库大小主要参考以下几点：
- 要加载的文件数量
- 每个文件中的数据量

对于非生产测试用途，建议使用较小的仓库
对于生产环境，使用较大仓库可能更具成本效益

由于snowflake是按秒计费，因此可以先从较大仓库开始，逐步调整到较小仓库

小而基本的查询，建议使用较小仓库，较大的仓库所拥有的额外计算资源不一定管用，因此并不一定更快
一般来说，应尽量使仓库的大小与要处理的查询的预期大小和复杂程度相匹配


# 在Snowsight使用仓库
Snowsight页面也会有一些操作会使用到仓库，比如说data preview
Snowsight页面执行查询通常使用的仓库大小是Small
通过客户端生成的语句一般不会产生费用，比如show table命令
可以在snowsight的monitor部分查看内部查询的运行



# 默认仓库与notebook apps
每个账户都配备了专为运行 Notebook Python 代码而设计的 SYSTEM$STREAMLIT_NOTEBOOK_WH仓库
该仓库有多群集，大小是X-small，其有助于减少群集碎片、优化成本并提高装箱效率

- 减少碎片化
  在大集群上执行轻量任务，会造成集群资源浪费，这种资源利用不满的问题，叫碎片化
  Notebook Python往往执行的是轻量的任务，这种轻量任务在小集群上执行，资源能得到更好的利用
- 优化成本
  小集群的计费粒度更细，自动扩缩到刚好需要的集群数，从而更好控制成本
- 提高装箱效率
  “装箱”是指把多个并发作业合理地“装”进计算资源的过程
  很多小盒子（小集群）比少数大盒子更容易把不同大小、时长的作业装得更满


# 仓库自动暂停(auto-suspend)
当仓库在一段时间内没有活动时，仓库可以设定自动暂停
这个时间段可以自定义，单位可以是分钟，小时等

建议设置一个相对短的时间段，因为snowflake是按秒收费的，比如5-10分钟或更短
短时间内没有查询，立刻暂停仓库，可以节省更多的credit

同时，也要考虑这个暂停时间段要和各查询之间的间隔相契合
比如，如果各查询的间隔是2~3分钟，则自动暂停的时间段就不能设置为1~2分钟，因为仓库会在每个查询结束之后不断启停，而每次恢复启动都会消耗credit(至少60秒)

**关闭自动暂停功能**
关闭仓库自动暂停功能，通常为以下情况：
- 仓库的工作负载大且稳定
- 仓库需要保持高可用，且提交查询后，不希望有延迟或滞后时间

一般来说，仓库启动后的准备过程会非常快（1或2秒），但如果仓库规格很大，则恢复仓库可能需要更长的时间


**注：**\
如果关闭了仓库的自动暂停功能，则需要考虑credit消耗
即便仓库在运行中没有处理任何查询，它依然会消耗credit
尤其当仓库的规格很大，比如X-Large, 2X-Large，credit消耗会很明显



关闭自动暂停功能，通过以下方式
- Web UI，选中仓库，设置Auto Suspend时，选择“Never”
- SQL，在创建或修改仓库时，把 AUTO_SUSPEND 参数设为 0 或 NULL

```sql
-- 修改已有仓库，禁用自动暂停
ALTER WAREHOUSE my_wh SET AUTO_SUSPEND = 0;

-- 或者
ALTER WAREHOUSE my_wh SET AUTO_SUSPEND = NULL;

-- 创建时直接禁用自动暂停
CREATE WAREHOUSE my_wh WITH WAREHOUSE_SIZE = 'XSMALL' AUTO_SUSPEND = 0;
```



# 仓库自动恢复(auto-resume)
仓库处于暂停状态时，如果有新的查询提交，仓库会被设定为自动恢复

关于是否启用仓库的自动恢复功能，可以通过以下几点考虑：
- 如果不需要考虑成本和访问权限的问题，则启用自动恢复以确保仓库在需要时启动，仓库的恢复可能会因配置而出现短暂延迟
- 如果需要考虑成本和访问权限的问题，则禁用自动恢复功能，仅在需要时手动恢复仓库

**关于仓库自动恢复和访问权限**\
如果AUTO_RESUME=TRUE，用户就可以提交查询，启动仓库，相当于用户可以访问仓库
如果AUTO_RESUME=FALSE，用户无法提交查询启动仓库，相当于用户被禁止访问仓库





# Scaling up与Scaling out
snowflake支持以下两种仓库扩展方式
- scaling up
  纵向扩展，调整仓库大小
- scaling out	
  横向扩展，调整仓库集群数量


# 仓库大小调整与仓库性能
调整仓库大小通常可以提高查询性能，特别是对于较大、较复杂的查询
如果仓库没有足够的计算资源来处理并发提交的所有查询，调整仓库大小还有助于减少排队现象
注：调整仓库大小虽然能部分缓解高并发问题，但最有效，最专业的方式是追加额外的仓库，或使用多集群仓库

调整仓库大小可以在任意时间点，包括仓库在运行中
如果查询运行缓慢，并且要在同一个仓库上运行具有相似大小和复杂程度的查询，可以选择在仓库运行时调整大小

调整仓库大小，需注意以下问题：
- 仓库规格越大，速度不一定越快
  对于已在快速执行的小型基本查询，调整大小可能不会有明显改善
- 仓库运行中调整大小不会影响已在处理的查询
  调整大小后，新增的额外计算资源在完全配置后，仅用于排队的查询和新查询
- 将 5XL 或 6XL仓库调整为4XL或更小的仓库会导致在旧仓库处于静默状态时，短时间内向客户同时收取新仓库和旧仓库的费用
  注：只有当从超大规格仓库 5XL 或 6XL 缩小到 4XL 或更小的规格时，才可能出现“旧仓库处于静默状态期间与新仓库短时并行，从而双计费”的情况
  从较小规格之间的缩容（例如 4XL→3XL、XL→L、2XL→XL 等）通常不会触发这种短时双计费机制

减小正在运行的仓库的大小会从仓库中删除计算资源
移除计算资源后，与这些资源关联的缓存也会被删除，这可能会影响性能，就像暂停仓库会在仓库恢复后会影响其性能一样，因此需要在节省credit和保持缓存之间权衡


# 多集群仓库与并发性
多集群仓库是专门用来处理针对多用户或查询条件下，查询的排队和性能问题的（多并发）
当并发数处于波动状态时，多集群仓库会自动处理应对
关于多集群仓库的使用以及集群的数量的设置，通过以下因素考虑：
- Enterprise版本下的仓库都是多集群仓库
- 一般情况下，都使用auto-scaling模式，只有特定情况下，使用maximized模式
- auto-scaling模式下，按照以下方式设置集群最小数和最大数
  - 最小数(minimum)
    一般情况下，保持使用默认值1，然后再根据需要启动额外的集群，如果考虑高可用，可以设置数值大于1
  - 最大数(maximum)
    在考虑仓库大小以及对应的credit消耗的前提下，尽可能多的设置最大值
