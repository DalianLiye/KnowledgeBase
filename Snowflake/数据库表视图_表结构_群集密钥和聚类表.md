当一个表，尤其是超大型表，不断发生DML时，该表的聚类情况可能随着时间推移会越来越糟\
可以手动重新排序数据，然后再重新插入数据到表中，操作繁琐，成本又高\
snowflake可以通过为表指定群集密钥来自动执行上记任务

定义了群集密钥的表被视为已聚类(clustered)\
群集密钥可以是表的一个或多个列/表达式

以下对象进行聚类
- 物化视图
- 表

不管是从一开始就对表进行聚类，还是后续持续对表进行聚类，都会带来额外的成本\
因此群集密钥并不适用于所有表

以下情况可以考虑使用聚类：
- 需要最快的响应速度，不计成本
- 查询性能的提高抵消了聚类和维护表所消耗的credit


# 群集密钥
群集密钥是表的一个或多个列或表达式，它们被显式指定用于将相似的行放在同一个微分区中\
这对于以下情况非常有用：
- 插入/加载数据时，导致排序不理想的表
- 大量DML导致表的自然聚类效果越来越差的超大型表

为表定义群集密钥的一般指标包括以下：
- 随着时间的推移，表上的查询运行速度慢于预期或明显下降
- 表的群集深度很大

定义群集密钥的方式
- create/alter table命令

表的群集密钥可以随时更改或删除\
不能为混合表定义群集密钥。在混合表中，数据总是按主键排序


# 群集密钥的优点
使用群集密钥有以下优点：
- 通过跳过与筛选谓词不匹配的数据，提高了查询的扫描效率。
- 表在定义群集密钥进行聚类后，其列压缩效果更好，尤其当其他列与群集密钥的列高度关联时
- 表在定义群集密钥后，未来对表中行进行的所有维护(目的是为了确保形成最佳群集)都将由 Snowflake 自动执行，无需额外管理，除非选择删除或修改该密钥


聚类可以显著提高性能并降低某些查询的成本，但执行聚类的计算资源也会消耗credit\
因此，只有当查询广泛受益于群集时，才应进行聚类

一般来说，如果查询经常按某(些)个列做 以下操作时，将其指定为表的群集密钥，Snowflake会把数据在存储层尽量按这个键的值集中放置\
因此，在执行查询时，Snowflake更容易只扫描相关的微分区，减少不必要的数据读取，从而提升性能
- WHERE 过滤
- ORDER BY 排序
- GROUP BY 分组
- JOIN

比如A表join B表，A表的join key设置为群集密钥，则可以快速找到对应的行



查询表的频率越高，聚类提供的好处就越多\
但是，表更改的频率越高，保持聚类的成本就越高\
因此，对于经常查询且不经常更改的表，聚类通常最具成本效益

表定义群集密钥后，行不一定会立即更新，Snowflake后台会自动判断，只有在表受益于操作时才执行自动维护\
也就是说，表在设置了群集密钥后，表中的数据行也不会立刻按照新的密钥重排，数据的重聚类是一个后台维护过程，不是实时发生的\
Snowflake会根据是否“有明显收益”来决定是否以及何时执行自动维护（比如重聚簇）\
当系统判断重排数据能减少扫描、提升查询性能、值得花费计算资源时，才会触发或优先进行这些维护操作\
如果当前收益不大，可能会推迟或不执行


# 为表选择群集的注意事项
当表满足以下所有条件，设置群集密钥是最佳选择：
- 该表包含大量的 微分区。通常，这意味着该表包含多兆字节 (TB) 的数据
- 查询可以利用群集的优势。通常，这意味着以下一项或两项是正确的 
  1) 查询是选择性的。换句话说，查询只需要读取表中一小部分行（因此通常是微分区的一小部分）
  2) 查询对数据进行排序（例如，查询包含一个ORDER BY子句）
- 群集密钥可以使很多查询收益。换句话说，许多/大多数查询都在相同的几列上进行选择或排序

简单总结就是，当一个表数据量很大，针对该表的查询类型要么是查询少量数据，要么查询时需要排序，并且有很多查询脚本都用到该表的相同几列用于排序或选择\
那么，这个表就可以放心大胆的配置群集密钥(配置为选择或排序时使用的那几个列)，因为好处多多

如果想降低整体成本，那么聚类表的使用模式应该是查询的次数很多，DML操作的次数要少\
如果非要对聚类表执行大量的DML操作，可以将DML操作分组分批次进行，这样DML整体次数就少了

在进行聚类之前，可以强烈建议对表执行一组代表性的查询，方便聚类后对比查看效果


# 选择群集密钥的策略
单个群集密钥可以包含一个或多个列或表达式\
对于大多数表，Snowflake建议每个密钥最多使用 3 列或 4 列（或者 3 个或 4 个表达式）\
添加超过 3-4 列往往会致使成本的增加大于收益

为群集密钥选择正确的列/表达式会显著影响查询性能\
对工作负载的分析通常会产生良好的群集密钥候选项

Snowflake 建议按以下顺序排列密钥的优先次序：
1) 将在选择性筛选器中最常用的列进行聚类\
   对于涉及基于日期的查询的多个事实表（例如“WHERE invoice_date > x AND invoice date <= y”），选择日期列是个好主意\
   对于事件表，如果有大量不同的事件类型，事件类型可能是一个不错的选择\
  （如果表中只有少量不同的事件类型，则在选择事件列作为群集密钥之前）
2) 如果有空间容纳其他群集密钥，则考虑联接谓词中经常使用的列，例如“FROM table1 JOIN table2 ON table2.column_A = table1.column_B”

如果通常根据两个维度（如 application_id 和 user_status 列）来筛选查询，那么在两个列上聚类可以提高性能\
列/表达式中非重复值（即基数）的数量是选择该列/表达式作为群集密钥的关键方面\
选择具有以下条件的群集密钥很重要：
- 数量足够多的非重复值可以在表中实现有效裁剪
- 数量足够少的非重复值允许 Snowflake 对同一微分区中的行进行有效分组

基数非常低的列可能只能进行最少的裁剪，例如仅包含布尔值的名为的 IS_NEW_CUSTOMER 列\
另一个极端是，基数非常高的列通常也不是直接用作群集密钥的良好候选项\
例如，包含纳秒时间戳值的列并非好的群集密钥



通常，如果列（或表达式）的基数更高，则在该列上保持群集的成本更高\
在唯一密钥上进行群集的成本可能超过在该密钥上进行群集的好处，尤其是在点查询不是该表的主要用例的情况下\
如果想使用基数非常高的列作为群集密钥，Snowflake 建议将该密钥定义为列上的表达式，而不是直接在列上定义，以减少非重复值的数量\
该表达式应保留列的原始顺序，以便每个分区中的最小值和最大值仍能进行裁剪\
例如，如果事实表的 TIMESTAMP 列 c_timestamp 包含许多离散值（远远超过表中微分区的数量），则可以通过将值转换为日期而不是时间戳（例如 to_date(c_timestamp)）来在该列上定义群集密钥\
这会将基数减少到总天数，这通常会产生更好的裁剪结果\
再举一个例子，可以使用 TRUNC 函数和刻度负值（例如 TRUNC(123456789, -5)）将一个数字截断为更少的有效数字

当为表指定多列群集密钥时，则在 CLUSTER BY 子句中指定列的顺序很重要\
一般而言，Snowflake 建议将列从 最低 基数到 最高 基数进行排列\
在较低基数列之前放置较高基数列通常会降低后一列的群集有效性

在文本字段上进行聚簇时，群集密钥元数据仅跟踪前几个字节（通常为 5 或 6 字节）\
对于多字节字符集，甚至会少于5个字符

在某些情况下，对 GROUP BY 或 ORDER BY 子句中使用的列进行群集可能会有所帮助\
但通常不如对在筛选器或 JOIN 操作中大量使用的列进行群集有用\
如果一些列在筛选器/联接操作中大量使用，而在 ORDER BY 或 GROUP BY 操作中使用不同的列，则优先使用筛选和联接操作中使用的列



# 重聚类
当对聚类表执行DML操作（INSERT、 UPDATE、 DELETE、 MERGE、 COPY）时，表中数据的聚类程度可能会降低\
需要定期对表进行重聚类以保持最佳群集

在重聚类过程中，Snowflake 使用聚类表的群集密钥来重新组织列数据，以便将相关记录重新定位到同一个微分区\
此 DML 操作会删除受影响的记录，然后重新插入这些记录，并根据群集密钥进行分组

Snowflake 中的重聚类是自动的；无需维护\
但是，对于某些账户，手动重聚类已被弃用，但仍允许使用


# 重聚类对credit和存储的影响
重聚类消耗的credit数量取决于以下：
- 表的大小
- 需要重聚类的数据量

重聚类会根据表的群集密钥对行进行物理分组，生成新的微分区，因此会产生存储成本\
即使在表中添加少量行也可能导致所有包含这些值的微分区被重新创建\
此过程可能会造成大量数据周转，因为原始微分区被标记为已删除，但会保留在系统中以启用Time Travel和故障安全

只有在 Time Travel 保留期和随后的故障安全期结束后（即如果使用的是 Snowflake Enterprise Edition [或更高版本]，Time Travel 最少延长 8 天，最多延长 97 天），原始微分区才会被清除\
这通常会导致存储成本增加

在为表定义群集密钥之前，应考虑相关的 credit 和存储成本


# 定义聚类表


## 计算表的群集信息
以下系统函数可以计算表的群集细节，包括群集深度
- SYSTEM$CLUSTERING_INFORMATION 

此函数可以在任何表上的任何列上运行，无论该表是否有显式群集密钥：
- 如果表有显式群集密钥，该函数只需表名
  ```sql
  SELECT SYSTEM$CLUSTERING_INFORMATION('my_table')
  ```
- 如果表没有显式聚簇键，或者虽然有聚簇键但想针对别的列（或表达式）来评估聚簇情况，那么调用时需要额外把这些列（或表达式）作为第二个参数传给函数
  ```sql
  SELECT SYSTEM$CLUSTERING_INFORMATION('my_table', '(col1, DATE_TRUNC(''day'', col2))');
  ```


## 为表定义群集密钥
创建表时，可以在create table命令的cluster by定义群集密钥，命令如下：
```sql
CREATE TABLE <name> ... CLUSTER BY ( <expr1> [ , <expr2> ... ] )
```
其中每个群集密钥由一个或多个表列/表达式组成，这些表列/表达式可以是任何数据类型，但GEOGRAPHY、 VARIANT、 OBJECT 或ARRAY除外

群集密钥可以包含以下任何内容：
- 基础列
- 基础列上的表达式
- VARIANT列中路径上的表达式

注：
- 对于每个 VARCHAR 列，目前执行聚类仅使用前 5 个字节\
  如果每行的前 N 个字符相同，或者没有提供足够的基数，则可以考虑在子字符串上进行聚类，该子字符串在相同字符之后开始，并且具有最佳基数
  ```sql
  create or replace table t3 (vc varchar) cluster by (SUBSTRING(vc, 5, 5));
  ```
- 如果将两个或更多列/表达式定义为表的群集密钥，则顺序会影响数据在微分区中聚类的方式
- 通过CREATE TABLE ...CLONE语句创建表时，会复制源表的群集密钥，但克隆的表的自动聚类是暂停的，必须手动恢复
- 通过CREATE TABLE...AS SELECT语句创建表时，它是不可以在语句中指定群集密钥的，但可以在创建表后再为表定义群集密钥
- 不支持直接使用VARIANT列定义为群集密钥，但是，如果从这个VARIANT列中提取出某个路径下的具体字段，并且把它转换成明确的数据类型，则可以用这个“表达式”来作为群集密钥


## 更改表的群集密钥
可以随时添加或更改表的群集密钥，命令如下：
```sql
ALTER TABLE <name> CLUSTER BY ( <expr1> [ , <expr2> ... ] )


ALTER TABLE t1 CLUSTER BY (c1, c3);  --群集密钥是c1和c3
ALTER TABLE T1 CLUSTER BY (SUBSTRING(C2, 5, 15), TO_DATE(C1));  --群集密钥变成了 LINEAR(SUBSTRING(C2, 5, 15), CAST(C1 AS DATE)) 
ALTER TABLE T1 CLUSTER BY (v:"Data":name::string, v:"Data":id::number);  --群集密钥变成了 LINEAR(TO_CHAR(GET_PATH(V, 'Data.name')), TO_NUMBER(GET_PATH(V, 'Data.id')))
```
注
1) 并不是所有的表达式都可以作为已填充表的群集密钥\
   可以通过show function命令来查看该function是否可以用在群集密钥
   ```sql
   show functions like 'function_name';  #返回结果有valid_for_clustering一列，用于标识该function是否可以用作已填充表的群集密钥
   ```
2) 更改表的群集密钥只是更改了表的聚类规则，表中数据并不会立刻按照新的密钥重新组织\
   只有当对这个表执行“重聚簇”（recluster）操作后，Snowflake才会根据新的聚簇键重聚类


## 删除表的群集密钥
可以随时删除表的群集密钥，命令如下：
```sql
ALTER TABLE <name> DROP CLUSTERING KEY
```
