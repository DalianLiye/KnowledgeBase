# Table的分类
根据数据的存储时间，Table可以分以下几种：
- Permanent Table\	
  永久表\
  用于存储需要长期存储的数据
- Temporary Table\	
  临时表\
  用于存储不需要长期存储的数据
- Transient Table\	
  瞬时表\
  存储不需要fail-safe的数据

注：hybrid table是不能创建为Temporary Table和Transient Table，即hybrid table必须是常驻的永久表\
hybrid tables 是按照什么分的？



# Temporary Table
Temporary Table存储的数据仅保存在session中，其他的user和session是无法查看的\
Session终止，则数据就会被清理，且无法恢复

注: 除了Table，一些其他的DB object也支持Temporary，比如stages

**关于Temporary Table的存储**\
Temporary Table里的数据也是会占用存储空间的，也会产生费用\
因此，当Temporary Table里的数据不再需要(尤其是数据量很大时)，最好在使用后删掉，或者断开session


**关于Temporary Table的维护**\
如果工作负载中生成大量临时表，则在 Information Schema中查询 COLUMNS视图或TABLES视图时，很可能会遇到性能下降的情况\
因为Temporary Table激增会增加系统维护的工作量，因此查询这些视图会导致性能下降

可以执行以下操作避免该问题：
- 在session结束之前，明确删除Temporary Table，不要非要等session结束后清理Temporary table
- 确保用户明确注销inactive session

一句话，就是Temporary Table或者session，不用的时候赶紧关闭


**关于Temporary Table与其他类型表的命名冲突**\
Temporary Table属于指定的Database和Schema下的指定session\
因此某一个session内，Temporary Table可以和Non-Temporary Table是同名的

session中，Temporary Table的优先级最高\
比如有一个Temporary Table叫tableA，也有一个Non-Temporary Table也叫TableA\
那么当sesion中对TableA做任何操作时，其实操作的都是Temporary Table的tableA，而不是Non-Temporary Table的TableA


**创建Temporary Table**
示例：
```sql
CREATE TEMPORARY TABLE mytemptable (id NUMBER, creation_date DATE);
```

注：\
创建Temporary Table不需要在schema上拥有CREATE TABLE权限\
Temporary Table创建后，是不能转换为其他表类型的



# Transient Table
Transient Table跟Permanent Table类似，只是它没有故障安全期\
比如删除表，它的数据在经过time travel之后，依然会保留7天，这个7天就是故障安全期\
因此，也不需要提供与Permanent Table相同级别的数据保护和恢复


**关于Transient Table的存储**\
Transient Table里的数据也是会占用存储空间的，也会产生费用\
由于Transient Table不需要故障安全保障，因此与Permanent Table不同，它不需要这方面的花费


**关于Transient Databases和Schemas**\
所有在Transient Schema下创建的Table，以及在Transient DB下创建的Schema都是Transient的


**创建Transient DB，Schema和Table**\
示例
```sql
CREATE TRANSIENT TABLE mytranstable (id NUMBER, creation_date DATE);
```


# 关于Time Travel和Fail-safe流程
1) 事务在T0时发生（例如删除了表中的行）。
2) 在T0到T0 + TimeTravel（例如 1 天/7 天/90 天）期间：\
   用户可以自行使用 Time Travel 恢复数据（SELECT ... AT, UNDROP, CLONE AS OF 等）。
3) TimeTravel结束, 并进入Fail-safe 期间（通常 7 天）：\
   数据不可通过 Time Travel 或 UNDROP 由用户访问或恢复\
   Snowflake 仍保留数据以便在极端情况下由 Snowflake 支持恢复
4) Fail-safe 结束，数据被永久删除且无法恢复



# Time Traval Retention Period
Time travel retention period是可以在表定义时被指定的\
在time travel retention period期间，可以做任何Time Travel 操作，包括数据恢复\
对于Permanent table, 如果Time travel retention period设置为0，则在删除时，它将立即进入Fail-Safe Period

Temporary Table的Time Travel保留期可为 1 天\
一旦session结束，Temporary Table就会被清除，因此实际保留期为24h或会话的剩余时间，以较短者为准

长时间运行的Time Travel查询会将Temporary Table和Transient Table的清除延后至查询完成


# Fail-Safe Period
Fail-Safe Period ,即故障安全期\
它不可以在表定义里面被指定，它是在snowflake内预定义的

Temporary Table和Transient Table是没有fail-safe period的，因此time travel retention 之后是不会产生任何费用的\
但数据经过变动且过了time travel retention period之后，数据也是无法被恢复的

注：
- 任何类型Table的fail-safe period均不可用户自行配置
- Temporary Table和Transient Table没有fail-safe period，因此在Time Travel之后不会产生额外的数据存储费用
