# 关于视图
视图允许像访问表一样访问查询结果

视图通过 CREATE VIEW命令创建
CREATE VIEW 命令可以使用完全限定、部分限定或不限定的表名
```sql
CREATE VIEW v1 AS SELECT ... FROM my_database.my_schema.my_table;
CREATE VIEW v1 AS SELECT ... FROM my_schema.my_table;
CREATE VIEW v1 AS SELECT ... FROM my_table;
```

视图用途，包括：
- 组合
- 隔离
- 保护数据


视图几乎可以在任何可以使用表的地方使用（连接、子查询等）

如果未指定schema，则Snowflake假定该表与视图处于相同的schema中
如果视图在定义时没有为表加上明确的数据库/schema前缀，那么在不同时间、不同会话或不同角色切换到不同的激活模式时，视图可能会引用到名字相同但位于不同schema中的表，从而导致视图在不同场景下指向“不同的表”


# 视图类型
Snowflake 支持两种类型的视图：
- 非物化视图（通常简称为“视图”）
- 物化视图


## 非物化视图
视图基本上是查询的命名定义
非物化视图的结果是执行视图定义里面的脚本返回的结果，结果不会被存储
非物化视图的性能比物化视图慢

任何返回有效结果的查询表达式都可以用于创建非物化视图
例如：
- 选择表中的部分（或全部）列
- 在表列中选择特定范围的数据
- 连接两个或多个表中的数据


## 物化视图
物化视图的结果是存储起来的，更像是表
物化视图可以加快访问速度，但需要存储空间和主动维护，这两者都会产生额外的成本
物化视图还有一些非物化视图没有的限制


## 安全视图
非物化视图和物化视图都可以定义为安全视图
与标准视图相比，安全视图有以下优势
- 改善数据隐私
- 改善数据共享

注：安全视图会有一些性能影响需要考虑


## 递归视图（仅限非物化视图）
非物化视图可以是递归的（即视图可以引用自身）
可以使用关键字 RECURSIVE 创建递归视图，而不是使用递归 CTE


# 视图的优点

## 视图允许编写更多模块化代码
视图可以将代码逻辑打包，模块化设计，可以被重用


## 视图允许授予对表的子集的访问权限
给用户授权视图的访问权限后，用户只能查看视图定义中表的部分字段
用户不能访问整个基表的数据

如果用户可以访问视图但不可以访问基表，视图的所有者role没有权限访问基表，则用户无法查询该视图
如果用户可以访问视图但不可以访问基表，视图的所有者role有权限访问基表，则用户可以查询该视图
如果用户可以访问视图也可以访问基表，不管视图的所有者role有没有权限访问基表，则用户都可以查询该视图


## 物化视图可以提高性能
物化视图的目的就是提高性能
物化视图包含表中数据子集的副本
根据表和物化视图中的数据量，扫描物化视图可能比扫描表快得多
物化视图还支持聚类，可以对同一数据创建多个物化视图，每个物化视图设置群集密钥，以便不同的查询可以在具有该查询的最佳群集的视图上运行


# 视图局限性
视图创建后无法更新，必须先删掉然后重新创建

视图是只读的，不能对视图执行 DML 命令
但可以在针对表的DML语句内，将视图用作子查询

对表定义的更改不会自动同步到在该表上创建的视图
比如，视图引用了一个表的某个字段，如果表的该字段被删除，那么视图定义里依然会引用这个字段
但由于这个字段已不存在，因此视图会变成invalid
