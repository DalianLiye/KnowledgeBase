# 身份认证与权限
身份认证是k8s确认请求者是谁的过程\
权限(RBAC相关资源)是k8s里的一种资源

任何操作外部的访问和内部的访问, 都需要认证和授权\
比如：kubectl工具或Dasboard向k8s发送命令，kubelet向API server发送请求

k8s中提供了良好的多租户认证管理机制，如RBAC，ServiceAccount还有各种策略等

通过查看控制平面节点的以下文件，可以查看RBAC访问控制的配置\
/usr/lib/systemd/system/kube-apiserver-service


## 认证
k8s集群有以下两类用户：
- Service Account(服务账户)，由k8s管理的
- User Account(普通账户)

Service Account和User Account的比较:
- 普通账户是针对(人)用户的，服务账户针对Pod进程
- 普通账户时全局性，在集群所有namespaces中，名称具有唯一性
- 通常，集群的普通账户可以与企业数据库同步，新的普通账号创建需要特殊权限，服务账户创建目的是更轻量化，允许集群用户为特定任务创建服务账户
- 普通账户和服务账户的审核注意事项不同
- 对于复杂系统的配置包，可以包括对该系统的各种组件的服务账户的定义


### UserAccount
User Account(普通账户)是假定被外部或独立服务管理的，比如Dashboard登录用账户\
User Account由管理员分配keys, User像使用keystore或google账号一样，被存储在包含usernames和passwords的list的文件里\
注: 在k8s中不能通过API调用将普通用户添加到集群中


### Service Account
Service Account通过以下控制器进行自动化管理：

- **Service Account Admission Controller**\
  通过Admission Controller插件来实现功能的

  Admission Controller是kube-apiserver的一部分，负责在Pod创建或更新时，会同步对Pod进行修改，自动为 Pod 注入与 Service Account 相关的配置，确保 Pod 能正确使用 k8s的身份认证和镜像拉取凭证
  
  当Admission Controller插件处于激活状态下（在大多数发行版中都默认情况）创建或修改Pod时，会按以下操作执行：
  
  1) 如果创建Pod时没有设置Service Account，则ServiceAccount将被设置为default，该default 账户就是Service Account Controller为每个namespace创建的default账户

  2) 确保Pod引用的Service Account存在，否则将会拒绝请求，最终导致Pod创建失败

  3) 如果Pod不包含任何imagePullSecrets, 则将Service Account的imagePullSecrets会添加到Pod中(前提是Service Account要有配置imagePullSecrets)，好处就是无需为每个Pod执行imagePullSecrets，只要跟Service Account绑定即可

  4) 为包含API访问的Token的Pod添加了一个volume\
     当Pod需要访问 Kubernetes API 时，是需要身份认证的，这通常通过挂载一个包含访问Token的 Secret来实现\
     Admission Controller会为Pod自动挂载一个包含Service Account Token的Secret，这样Pod内部的应用就可以通过该Token访问API了\
     Admission Controller 会自动创建一个名为 kubernetes.io/service-account-token 类型的Secret，并将该Secret以Volume的形式挂载到Pod中

  5) 把volumeSource添加到Pod的每个容器中\
     挂载路径：/var/run/secrets/kubernetes.io/serviceaccount

 - **Token Controller**\
   Token Controller主要管理和维护与Service Account Token相关的 Secret 对象

   **主要作用:**
   - 创建Service Account时，自动创建Service Account Token Secret
   - 监控Service Account的变化，并自动刷新和更新Token
   - 删除Service Account时，清理过期或无用的 Token Secret
    
- **Service Account Controller**\
  Service Account Controller主要用来管理namespaces下的ServiceAccount，它会确保每个有效的namespaces中都存在一个名为"default"的ServiceAccount

  Pod运行时需要通过Service Account来访问Kubernetes API

  默认情况下，如果用户没有显式指定Pod使用哪个Service Account，系统会自动将其设置为 default，从而避免Pod没有Service Account 导致无法访问 API 或认证失败


Service Account相关命令：
```shell
kubectl get sa -n <namepsace>
kubectl get serviceaccount -n <namepsace>
```

## 授权
k8s的授权都是基于角色来进行控制访问的\
即RBAC， Role Based Access Control


### Role
Role代表了一个角色，它包含了一组权限

关于Role包含的权限，有以下特点:
- 只定义允许(Allow)规则，即明确列出哪些资源和操作是被允许的
- 不定义拒绝(Deny)规则，即没有类似“禁止”或“拒绝”的规则
- 默认拒绝：如果某个操作没有被明确允许，则该操作就会被拒绝

Role是namespace级别的资源，只能作用与namespace之内

查看Role信息，执行命令：
```shell
kubectl get role -n <namespace> <role_name> -o yaml
```

### ClusterRole
功能与Role一样，区别是资源类型为集群类型，而Role只作用在Namespace

查看Cluster_Role信息，执行命令：
```shell
kubectl get clusterrole <cluster_role_name> -o yaml
```

### RoleBinding
Role只是用于指定权限集合，具体作用与什么对象上，需要使用RoleBinding来进行绑定\
RoleBinding是命名空间级别资源，可以将Role或ClusterRole绑定到集群的User和Group上，或者某个命名空间的Service Account上

查看RoleBinding信息
```shell
kubectl get rolebinding -A
```

查看指定RoleBinding的配置信息
```shell
kubectl get rolebinding <role_binding_name> -A -o yaml
```

### ClusterRoleBinding
ClusterRoleBinding是集群级别资源\
ClusterRoleBinding的功能与RoleBinding一样，只是作用的对象是集群级别的资源



### RoleBinding和ClusterRoleBinding比较

- **RoleBinding**
  - **作用范围**\
    RoleBinding所在命名空间

  - **绑定的角色**	
    - Role（命名空间角色）
    - ClusterRole（集群角色）

  - **绑定的对象**
    - User/Group（集群范围）
    - ServiceAccount（同一命名空间）

  - **权限生效范围**\
    RoleBinding所在命名空间
  
  - **使用场景**\
    给某个命名空间内的User或Service Account授权


- **ClusterRoleBinding**
  - **作用范围**\
    整个集群

  - **绑定的角色**	
    - ClusterRole（集群角色）
  
  - **绑定的对象**
    - User/Group（集群范围）
    - ServiceAccount（任一命名空间）

  - **权限生效范围**\
    整个集群

  - **使用场景**\
    给集群范围内的User、Group或某个命名空间的Service Account授权
